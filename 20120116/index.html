What is Design Pattern? A general reusable solution
to a commonly occurring problem
within a given context in software design
Examples:
  • Null object:   avoid null references by providing a default object
  • KISS:   "keep it short and simple"
In other words: BEST PRACTICES

Anti Pattern

What is Anti Pattern?
pattern that may be commonly used
but is ineffective
and/or counterproductive in practice
Anti Pattern Categories
- Organizational anti-patterns
 - Project management anti-patterns
 - Analysis anti-patterns
 - Software design anti-patterns
 - Object-oriented design anti-patterns
 - Programming anti-patterns
 - Methodological anti-patterns
 - Configuration management anti-patterns
Organizational anti-patterns
Analysis paralysis: Analysis is taking more time than development
Design by committee: Too many contributors, no unifying vision
Escalation of commitment: Doesn't revoke wrong decision
Management by perkele: Authoritarian style of management with zero tolerance
Management by objectives: Management that focus exclusively on quantitative management criteria
Moral hazard: I decide, you responsible
Mushroom management: Keeping developers uninformed and misinformed
Stovepipe or Silos: A structure that supports up-down flow of data but inhibits cross organizational communication

Project management anti-patterns
Groupthink: Avoid promoting viewpoints outside the comfort zone during groupthink
Overengineering: Spending resources making a project more robust and complex than is needed
Smoke and mirrors: Demonstrating unimplemented functions as if they were already implemented
Software bloat: Allowing successive versions of a system to demand ever more resources
Waterfall model: An older method of software development that inadequately deals with unanticipated

Software design anti-patterns
Abstraction inversion: Re-implement it using higher level functions
Ambiguous viewpoint: Presenting a model without specifying its viewpoint
Big ball of mud: A system with no recognizable structure
Database-as-IPC: Always use database as the ipc
Gold plating: extra feature without adding value
Input kludge: Failing to handle invalid input
Interface bloat: Making an interface so powerful that it is extremely difficult to implement
Magic pushbutton: Coding implementation logic directly within interface code
Race hazard: Failing to see the consequence of different orders of events
Stovepipe system: A barely maintainable assemblage of ill-related components

Object-oriented design anti-patterns
Call super: Requiring subclasses to call a superclass's overridden method
Circular dependency: Unnecessary mutual dependencies between software modules
God object: Concentrating too many functions in a single part of class
Object cesspool: Reusing objects whose state does not suitablefor re-use
Object orgy: All public !!!
Input kludge: Failing to handle invalid input
Poltergeists: Objects whose sole purpose is to pass information to another object
Sequential coupling: A class that requires its methods to be called in a particular order
Yo-yo problem: A structure (e.g., of inheritance) that is hard to understand due to excessive fragmentation

Programming anti-patterns
Accidental complexity: Unecessary complexity
Action at a distance: Unexpected interaction between parts of a system
Blind faith: Lack of checking of the result of a subroutine
Boat anchor: Retaining unused part of a system
Busy waiting: Consuming CPU while waiting for something to happen, usually by repeated checking instead of messaging
Caching failure: Not resetting a flag when it has been changed
Cargo cult programming: Using patterns and methods without understanding why
Coding by exception: Adding new code to handle each special case as it is recognized
Error hiding: Showing nothing or showing a meaningless message

Programming anti-patterns
Hard code: Embedding assumptions about the environment of a system in the code
Lava flow: Retaining undesirable code
Loop-switch sequence: Conditions in a loop
Magic numbers: Unexplained numbers in algorithms
Magic strings: Literal strings in code, for comparisons, as event types etc.
Repeating yourself: Writing code which contains repetitive patterns and substrings over again
Soft code: Storing business logic in configuration files rather than source code
Spaghetti code: Programs whose structure is barely comprehensible, especially because of misuse of code structures
Shotgun surgery: Developer adds multiple features to codebase in a single change

Methodological anti-patterns
Copy and paste programming Golden hammer, Silver bullet: Assuming that a favorite solution is universally applicable
Improbability factor: Assuming that it is improbable that a known error will occur
Not Invented Here (NIH) syndrome,Reinventing the wheel,Reinventing the square wheel: Failing to adopt an existing, adequate solution
Premature optimization: Coding early-on for perceived efficiency, sacrificing good design, maintainability, and sometimes even real-world efficiency
Programming by permutation (or "programming by accident"): Trial and error
Tester Driven Development: Software projects in which new requirements are specified in bug reports

Configuration management anti-patterns

Dependency hell: Problems with versions of required products
